# AGENTS.md

本文件用于规定员工工时管理系统项目的整体开发规则。后续在本项目中进行需求分析、架构设计、代码编写、页面设计、数据库设计、测试、部署和文档补充时，都必须优先遵循本文件。

## 项目目的

本项目的目标是设计并开发出一个完整的员工工时管理系统，并最终达到可以部署上线、可以持续维护、可以版本更迭的系统平台标准。

系统需要解决企业日常工时管理中纸质表单和零散电子表格带来的问题，包括工时数据不统一、审批流程不透明、项目授权不清晰、统计分析效率低等。系统应通过信息化方式完成基础数据维护、员工项目授权、员工工时填报、部门经理审批、个人/部门/公司工时查询统计和操作日志追踪。

系统建设完成后，应具备以下能力：

- 管理员可以维护部门、用户和项目等基础数据。
- 部门经理可以为本部门员工分配项目并审批本部门员工工时。
- 员工可以填报、修改、删除、提交和查询本人已授权项目的工时。
- 系统可以按照个人、部门和公司维度进行工时查询统计。
- 工时申报单的关键操作可以通过日志追踪。
- 系统可以独立部署上线，并支持后续功能扩展和版本迭代。

工作职责包括：

- 根据需求分析文档理解业务规则。
- 根据数据库设计文档和 ER 图约束数据结构。
- 根据 `DESIGN.md` 约束前端 UI 风格。
- 设计前后端分离架构和模块划分。

- 编写必要的接口、页面、数据库脚本和部署说明。
- 发现需求矛盾时先指出问题，再给出推荐方案。

## 技术栈

当前项目暂定技术栈如下：

- 后端：java + spring boot

- 数据库：mysql

- 常用开发工具：intellij idea 2025.3.3
- 常用数据库工具：navicat 17

开发时优先使用用户电脑中最适合当前任务的软件和命令行工具。若需要操作后端项目，优先考虑 intellij idea 项目结构和 maven/gradle 构建方式；若需要查看或设计数据库，优先考虑 navicat 17 和 mysql 语法；若需要运行前端，优先使用项目自带的包管理器和启动脚本。

除非用户明确要求，不随意更换核心技术栈，不擅自引入大型替代框架。

## 开发流程

开发时必须按步骤推进，不允许没有分析就直接大面积生成代码。每完成一个关键步骤后，需要向用户说明当前结果，并询问用户是继续下一步还是需要修改当前步骤。

推荐流程如下：

### 第一步：确认目标

需要先明确本轮任务要完成什么，例如：

- 新建后端项目。
- 新建前端项目。
- 设计数据库表。
- 编写登录功能。
- 编写部门管理功能。
- 编写工时填报功能。
- 编写部署说明。

完成本步骤后，需要向用户确认：当前理解是否正确，是否继续进入下一步。

### 第二步：阅读依据

根据任务读取对应资料：

- 业务功能读取需求分析文档。
- 数据表设计读取数据库设计文档。
- 页面样式读取 `DESIGN.md`。
- 代码修改读取已有项目结构。

完成本步骤后，需要总结读取到的关键规则，并询问用户是否继续。

### 第三步：设计方案

在编码前先给出实现方案，包括：

- 涉及哪些页面。
- 涉及哪些接口。
- 涉及哪些数据库表。
- 涉及哪些后端类。
- 涉及哪些前端组件。
- 数据如何流转。
- 权限如何校验。
- 异常如何处理。

完成本步骤后，需要询问用户是否按该方案继续实现。

### 第四步：创建或修改文件

按照确认后的方案创建或修改文件。修改时必须遵循已有项目结构和命名风格。

完成本步骤后，需要说明：

- 新增了哪些文件。
- 修改了哪些文件。
- 每个文件大概负责什么。

然后询问用户是否继续测试或调整。

### 第五步：运行和测试

根据任务类型执行测试：

- 后端功能需要运行单元测试或接口测试。
- 前端页面需要启动项目并检查页面效果。
- 数据库变更需要检查建表语句和约束。
- 涉及联调时需要验证前后端接口数据是否一致。

完成本步骤后，需要说明测试结果，并询问用户是否继续修复问题或进入下一项功能。

### 第六步：交付总结

交付时需要说明：

- 本次完成了什么。
- 主要文件在哪里。
- 如何运行或验证。
- 还剩下什么后续工作。

如果本次任务没有完全完成，需要明确说明原因和下一步建议。

## 代码风格

代码使用正常、清晰、易读的工程风格。因为本项目最后输出是为了让用户学习，所以新增的手写业务代码需要尽量做到每行或每个关键语句都有中文注释，帮助用户理解代码作用。

代码要求如下：

- 命名必须清晰，避免无意义缩写。
- java 类名使用大驼峰命名，例如 `UserController`。
- java 方法名和变量名使用小驼峰命名，例如 `getUserById`。
- 数据库表名和字段名使用小写英文加下划线，例如 `work_time_apply`。
- 前端文件和目录使用小写英文命名，必要时使用短横线分隔。
- 接口路径使用小写英文和短横线或复数资源名。
- 不写难懂的炫技代码，优先写适合学习和维护的代码。
- 不把多个复杂逻辑挤在一行。
- 不在代码中硬编码数据库密码、密钥、服务器地址等敏感信息。
- 重要业务判断必须写清楚注释，例如权限校验、状态流转、删除限制、日志记录。

注释要求如下：

- 新增业务代码尽量逐行添加中文注释。
- 配置文件中的关键配置项需要添加说明。
- SQL 脚本中的表和关键字段需要添加注释。
- 前端页面中的关键交互逻辑需要添加注释。
- 对于框架自动生成、简单 getter/setter、import 语句等机械代码，不强行写无意义注释。

## 架构规则

项目采用前后端分离架构：

- 前端负责页面展示、表单交互、基础输入校验和接口调用。
- 后端负责业务规则校验、权限判断、状态流转、数据持久化和接口响应。
- 数据库负责保存结构化数据。
- 前后端通过 REST 风格接口交互。
- 后端接口返回格式必须统一。

后端建议分层：

- controller：接收请求、校验基础参数、调用 service。
- service：处理业务规则、权限判断、状态流转和事务。
- mapper/dao：访问 mysql 数据库。
- entity/model：封装数据库表对象。
- dto：封装接口入参。
- vo：封装接口返回结果。
- config：放置项目配置。
- exception：放置统一异常处理。

课程文档或时序图中可能采用“控制器直接调用数据访问层”的简化画法，但实际编码时建议保留 service 层，保证项目能长期维护和部署上线。

## 前端 UI 规则

前端 UI 必须严格遵循 `DESIGN.md`。设计页面、组件、颜色、间距、字体、圆角、按钮、表格、输入框、卡片和弹窗时，以 `DESIGN.md` 为最高优先级规范。

重点要求如下：

- 整体风格采用接近 Notion 的安静、文档化、轻量办公风格。
- 页面主画布优先使用暖白色 `#f6f5f4`。
- 内容区域使用白色或接近白色的 surface。
- 主操作色使用 `#0075de`。
- 按下或激活态使用 `#005bab`。
- 不随意新增第二套主色。
- 表格清晰、克制、适合业务系统扫描阅读。
- 输入框保持小圆角，不使用夸张胶囊形。
- 工时管理系统是业务系统，不做营销式落地页，不做过度装饰。

常见页面要求如下：

- 登录页：简洁认证卡片，手机号、密码、登录按钮、错误提示风格统一。
- 管理列表页：筛选区、表格、分页、操作按钮清晰。
- 新增/编辑页：表单布局清楚，校验提示明确。
- 统计页：使用信息密度适中的卡片、表格和图表。
- 弹窗/确认框：白色背景、轻阴影、按钮简洁。

## 业务角色规则

系统用户统一使用 `user` 表示，普通员工、部门经理和管理员通过 `user_role` 区分。

角色取值如下：

- `0` 表示管理员。
- `1` 表示部门经理。
- `2` 表示员工。

权限规则如下：

- 管理员负责部门、用户、项目等基础数据管理，并可查看公司级工时数据。
- 部门经理负责本部门员工项目分配、本部门员工工时审批、部门工时查询统计。
- 员工负责本人已授权项目的工时填报、修改、删除、提交审批和个人工时查询统计。
- 员工只能填报已授权项目的工时。
- 部门经理只能管理和审核本部门范围内的数据。
- 不允许跨部门项目授权。
- 后端必须做权限校验，不能只依赖前端隐藏按钮。

## 数据库设计规则

数据库表结构以最终 ER 图和数据库设计文档为准。当前核心表包括：

- `department`：部门表。
- `user`：用户表。
- `project`：项目表。
- `user_project`：用户项目参与表。
- `work_time_apply`：工时申报单表。
- `work_time_log`：工时操作日志表。

数据库规则如下：

- 所有表名和字段名统一使用小写英文和下划线。
- 主键字段建议使用 `bigint` 并自增。
- 外键字段命名与关联主键保持一致，例如 `dept_id`、`user_id`、`project_id`、`work_id`。
- 枚举字段使用 `tinyint` 表示。
- 文档和代码中必须明确枚举字段取值。

枚举字段规则如下：

- `user_role` 取值为 `0`、`1`、`2`，分别表示管理员、部门经理、员工。
- `project_status` 取值为 `0`、`1`，分别表示禁用、启用。
- `auth_status` 取值为 `0`、`1`，分别表示取消授权、有效授权。
- `work_time_apply.status` 取值为 `0`、`1`、`2`、`3`，分别表示草稿、待审批、审批通过、已驳回。
- `operation_type` 取值为 `0`、`1`、`2`、`3`、`4`，分别表示新建、修改、提交、审批通过、驳回。

`user_project` 中的 `user_id` 和 `project_id` 在逻辑上组合唯一，表示同一用户与同一项目只能存在一条参与记录。表结构文档中可以分开列出两个字段，但实现时需要通过唯一索引或业务校验保证该约束。

## 工时申报状态规则

工时申报单状态由 `work_time_apply.status` 表示。

状态取值如下：

- `0` 草稿：用户新建工时申报单后，尚未提交审批。
- `1` 待审批：用户提交后，等待部门经理审核。
- `2` 审批通过：部门经理审核通过，工时可进入统计。
- `3` 已驳回：部门经理审核不通过，用户可修改后重新保存并提交。

状态流转规则如下：

- 草稿可以提交为待审批。
- 待审批可以审批通过。
- 待审批可以驳回为已驳回。
- 已驳回可以修改后回到草稿。
- 审批通过后的工时原则上不得随意修改。

提交时间、审批时间、驳回时间、修改时间等操作时间统一记录在 `work_time_log` 中，不放入 `work_time_apply` 作为普通字段。

## 核心业务规则

- 同一用户、同一项目、同一工作日期只能存在一条有效工时申报单。
- `work_date` 不能晚于当前日期。
- `work_hours` 按 0.5 小时粒度填写。
- `work_hours` 范围建议为 0.5 到 24 小时。
- 员工只能选择有效授权项目填报工时。
- 项目禁用后不允许新增授权或新增工时填报。
- 删除用户或项目前，必须检查是否存在相关工时申报数据。
- 存在工时申报数据的用户或项目不允许直接物理删除。
- 工时提交、修改、审批通过、驳回等关键操作必须写入工时操作日志。
- 部门经理审批驳回时应填写驳回原因。
- 查询统计应根据用户角色控制数据范围。

## 接口规则

接口命名围绕资源设计，推荐路径如下：

- `/api/auth/login`
- `/api/users`
- `/api/departments`
- `/api/projects`
- `/api/user-projects`
- `/api/work-times`
- `/api/work-time-logs`
- `/api/statistics`

接口实现要求如下：

- 所有需要登录的接口必须校验用户身份。
- 管理员、部门经理、员工接口必须校验角色权限。
- 涉及部门范围的数据必须校验 `dept_id`。
- 写操作必须校验参数合法性和业务规则。
- 异常信息应对用户友好，不直接暴露数据库异常细节。
- 密码不得明文存储，实际实现时应使用安全哈希。
- 返回结构统一，至少包含状态码、提示信息和数据内容。

- 不允许只在前端做权限控制，后端必须再次校验。
- 不允许直接物理删除存在历史工时数据的用户或项目。
- 不允许忽略工时操作日志。
- 不允许为了赶进度写难以理解、难以维护的代码。
- 不允许引入用户没有要求且项目不需要的大型框架。
- 不允许在没有说明影响范围的情况下进行大面积重构。

## 安全与部署要求

本项目目标是可以部署上线，因此开发时必须考虑安全和部署。

安全要求如下：

- 密码使用哈希方式存储。
- 登录接口需要基础失败处理。
- 后端接口必须进行权限校验。
- 关键写操作必须记录日志。
- 参数必须校验，避免非法数据进入数据库。
- 生产环境不暴露详细错误堆栈。
- 数据库账号、密码、jwt 密钥等敏感信息不能硬编码。
- 上线时建议使用 https。

部署要求如下：

- 后端能够打包为可运行服务。
- 前端能够构建为静态资源。
- 数据库提供建表脚本和初始化数据。
- 项目提供部署说明。
- 开发、测试、生产环境配置应区分。
- 系统应支持后续版本升级和数据库变更。

## 完成标准

本项目的最终完成标准不是只写几个页面或接口，而是形成一个能部署上线且能版本更迭的系统平台。

最低完成标准如下：

- 前端页面可以正常访问。
- 后端接口可以正常启动。
- mysql 数据库表结构完整。
- 登录功能可用。
- 管理员基础数据管理可用。
- 部门经理项目授权和工时审批可用。
- 员工工时填报、修改、删除、提交和查询可用。
- 个人、部门、公司工时查询统计可用。
- 工时状态流转正确。
- 工时操作日志可追踪。
- 权限控制基本正确。
- 项目可以打包部署。
- 项目有基本的运行说明和部署说明。

更高完成标准如下：

- 代码结构清晰，便于后续维护。
- 前端 UI 与 `DESIGN.md` 保持一致。
- 接口返回格式统一。
- 异常处理统一。
- 数据库约束合理。
- 有初始化数据。
- 有测试记录。
- 有版本迭代空间。

## 修改原则

- 优先保持需求文档、ER 图、状态图、数据库设计和代码实现一致。
- 若发现需求矛盾，应先指出矛盾，再给出推荐改法。
- 新增功能前先确认模块、接口、页面和数据库影响。
- 修改已有功能前先说明可能影响范围。
- 每完成一个阶段性步骤，需要询问用户继续还是调整。
- 所有输出都应尽量让用户能看懂、能学习、能复现。
